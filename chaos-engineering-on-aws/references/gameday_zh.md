# Game Day 主持人指南

## 概述
Game Day 是一种结构化的团队演练，通过模拟真实故障来验证韧性、事件响应流程和团队准备情况。

## 准备阶段（Game Day 前 1-2 周）

### 场景选择
1. 从步骤 1 的实验目标中选择 1-3 个高影响场景
2. 确保所有场景都已单独验证过（Game Day 中不使用未测试的实验）
3. 按复杂度评级：绿色（单服务）、黄色（多服务）、红色（跨 AZ/区域）

### 通信模板

#### 公告邮件/消息
主题：[Game Day] {日期} — EKS 韧性演练

团队，

我们将在 {日期} {开始时间} 至 {结束时间} 进行韧性 Game Day。

**目的**：验证 {应用名称} 的事件响应和系统韧性。
**范围**：{场景摘要}
**影响**：{预期影响范围} — 生产流量不会受影响 / 将部分受影响
**准备工作**：请在演练前查看 {运维手册链接} 处的运维手册。

有问题请联系 {主持人姓名}。

#### 事件频道设置
创建专用频道：#gameday-{日期}-{应用名称}
置顶以下内容：
- 场景说明文档
- Kill Switch 命令
- 紧急升级联系人
- 运维手册链接

### 角色卡

| 角色 | 职责 | 所需技能 | 准备材料 |
|------|------|---------|---------|
| **主持人** | 推进议程、管理时间、注入场景更新 | 高级 SA / SRE | 本指南、场景、计时器 |
| **事件指挥官** | 协调响应、做出升级决策 | 高级 SRE / 值班负责人 | 运维手册、升级矩阵 |
| **混沌操作员** | 执行故障注入、管理 Kill Switch | 熟悉 FIS/CM 的工程师 | FIS 控制台访问权限、脚本 |
| **记录员** | 记录每个操作和时间戳 | 任何团队成员 | 共享文档、模板 |
| **观察员** | 不干预观察，记录流程差距 | 管理层 / 外部 SA | 评分卡 |

### Game Day 前检查清单
- [ ] 所有参与者已确认且角色已分配
- [ ] 场景已在预发布/预生产环境中单独验证
- [ ] Kill Switch 已测试并记录
- [ ] 监控仪表板已准备并共享
- [ ] 通信频道已创建
- [ ] 运维手册已审查且可访问
- [ ] 基线指标已采集（等同阶段 0）
- [ ] 已获得管理层批准（如在生产环境）

## 执行日议程

### 议程模板（3 小时）

| 时间 | 活动 | 负责人 | 备注 |
|------|------|--------|------|
| 00:00-00:15 | 启动：回顾场景、确认 Kill Switch、角色 | 主持人 | 屏幕共享场景文档 |
| 00:15-00:30 | 基线检查（阶段 0） | 操作员 | 验证所有系统正常 |
| 00:30-01:30 | 故障注入 + 团队响应（阶段 1-3） | 操作员 + 事件指挥官 | 记录员记录所有操作 |
| 01:30-01:45 | 停止注入、确认完全恢复（阶段 4） | 操作员 | 验证基线恢复 |
| 01:45-02:30 | 热复盘 | 主持人 | 使用以下结构化问题 |
| 02:30-03:00 | 行动项 + 总结 | 主持人 | 每项 → 负责人 + 截止日期 |

### 热复盘问题
1. 演练中有什么让你感到意外？
2. 我们的监控有哪些盲区？
3. 运维手册是否充分？缺少了什么？
4. 故障检测花了多长时间？是否足够快？
5. 升级路径是否清晰？
6. 如果这是高峰流量期间的真实事件，会发生什么？

### 评分卡

| 维度 | 评分 (1-5) | 备注 |
|------|-----------|------|
| **检测速度** | | 故障被检测到的速度？ |
| **定位准确性** | | 根因是否被正确识别？ |
| **沟通效果** | | 更新是否及时且清晰？ |
| **运维手册质量** | | 运维手册是否覆盖了该场景？ |
| **恢复时间** | | 实际 MTTR vs 目标 RTO |
| **团队协作** | | 角色是否顺畅运作？ |
| **工具有效性** | | 监控/告警工具是否充分？ |
| **整体准备度** | | 团队整体准备情况 |
